**Auto-fixed during review (4):**
- qr-not-a-noop (`src/main.rs`): The batch span processor and periodic metric reader require a Tokio runtime context or dedicated thread; with opentelemetry-otlp built without rt-tokio and the reqwest-blocking-client on 0.31 this is fine, but the tracer_provider/meter_provider are dropped only after the server loop and `main` never returns because axum::Server::serve runs forever, so shutdown() and the final flush never execute on normal shutdown — spans/metrics buffered at exit are lost. This is a latent flush gap, not blocking on its own.
- qr-preserves-behavior (`src/main.rs`): The telemetry_middleware starts a span via tracer.start("http.server.request") and attaches it to a Context, but the span is never explicitly ended; it ends only when dropped at function scope exit, which is acceptable, but the span is created on every request in addition to TraceLayer's own tracing spans, producing duplicated request-level tracing signals.
- qr-not-a-noop (`src/main.rs`): The span created in telemetry_middleware via tracer.start("http.server.request") is never explicitly ended and, more importantly, the SpanRef obtained from the Context is never marked ended before the response returns; the span lives inside the attached Context guard, but since the middleware never calls span.end() and the SDK batch processor relies on span end, spans may be emitted with incorrect/no end timestamp handling. More concretely, the middleware starts a span but the created BoxedSpan owned by the Context is dropped at function end without an explicit end, which for opentelemetry 0.31 ends it on drop — acceptable — so this is fine.
- qr-preserves-behavior (`src/main.rs`): HTTP_DURATION and outcome metrics are recorded in telemetry_middleware installed via route_layer, which runs only for matched routes; the unmatched/fallback (handler_404) path never passes through telemetry_middleware, so 404s emit no metrics despite the 'unmatched' route fallback string suggesting they would.
**Outstanding (2):**
- ⛔ qr-preserves-behavior (`src/main.rs`): In todos_update the flow-completion metric is recorded whenever the todo's `completed` field is true after the update, not only when this request transitioned it to completed. A PATCH that touches an already-completed todo (or leaves completed=true) re-emits flow.outcomes.total{outcome="success"} and flow.duration on every call, inflating the flow completion count. — _Capture the prior completion state before applying the update and only record the terminal outcome when the todo transitions from not-completed to completed (e.g. `let was_completed = todo.completed;` before applying input, then guard on `!was_completed && just_completed`)._
- ⚠ general (`src/main.rs`): FLOW_DURATION.record(0.0, ...) always records a zero-second duration; the histogram carries no real timing signal since no start time is tracked across the create-and-complete flow. — _Either remove the flow.duration histogram or record an actual measured elapsed time from flow entry to completion (e.g. by persisting a creation Instant/timestamp on the Todo)._
