**Auto-fixed during review (3):**
- qr-preserves-behavior (`src/main.rs`): The router calls both `.with_state(db)` and `.with_state(telemetry)` with two different state types (Db and Telemetry), but axum 0.6's `Router<S>` is generic over a single state type. Handlers extract both `State<Db>` and `State<Telemetry>`, which axum 0.6 cannot satisfy with a single state — this will not compile (mismatched/incomplete state type).
- qr-not-a-noop (`src/main.rs`): The OTLP metric and span exporters are built with `.with_http()` using the reqwest-blocking-client feature, but the metric exporter is fed into `.with_periodic_exporter` (a background thread using a blocking client is acceptable) while the span exporter uses `.with_batch_exporter`; under a fully async tokio runtime the batch/periodic readers require a compatible runtime feature. More critically the SpanExporter with reqwest-blocking-client inside `with_batch_exporter` on the tokio runtime can panic/deadlock, preventing telemetry export.
- qr-preserves-behavior (`src/main.rs`): meter_provider.shutdown() and tracer_provider.shutdown() are placed after axum::Server::serve().await.unwrap(); since serve() runs forever (until process kill or panic), these shutdown/flush calls are unreachable on normal operation and buffered telemetry is never flushed on shutdown.
